# Naming & Systemic Memory Protocol

> **Purpose:** To establish permanent standards for file/directory naming, safety mechanisms, and continuous systemic learning to prevent architectural degradation.

## 1. Global Naming Protocol
- **Short & Intuitive:** Always use the most concise, universally understood tech terms (e.g., `app`, `db`, `api`, `scripts`). Avoid long descriptors like `shared_scripts` or `praem-os-ui`.
- **Formatting:** 
  - Strictly lower case.
  - No spaces. Use `snake_case` or `kebab-case` only.
- **Safety Rule:** **NEVER** perform bulk file or directory renaming without executing an explicit "Search & Replace" phase to update all internal Markdown and script links.

## 2. The 3-Tier Systemic Memory Activation
This system learns from failure. If an agent or workflow fails, these three tiers must be activated immediately:
1. **Tier 1 (Feedback Loop):** All architectural mistakes, failed attempts, and the resulting corrections MUST be logged immediately in `system/session_log.md` (both English & Thai vaults).
2. **Tier 2 (SOP Playbooks):** Structural or behavioral rules born from mistakes must be codified into this playbook or a new one in `system/skills/`. All agents are expected to operate under these standards.
3. **Tier 3 (HR Audits):** The Chief People & Performance Officer (Grace) will continuously audit agent workflows. Grace ensures adherence to the HITL (Human-in-the-loop) policies, token efficiency (no unnecessary loops), and compliance with the Naming Protocol.
