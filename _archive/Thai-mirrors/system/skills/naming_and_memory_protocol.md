# กฎมาตรฐานการตั้งชื่อและหน่วยความจำระบบ (Naming & Systemic Memory Protocol)

> **จุดประสงค์:** เพื่อสร้างมาตรฐานถาวรสำหรับการตั้งชื่อไฟล์/โฟลเดอร์, กลไกความปลอดภัย, และการเรียนรู้ระบบอย่างต่อเนื่องเพื่อป้องกันความเสื่อมถอยของสถาปัตยกรรม

## 1. Global Naming Protocol (มาตรฐานการตั้งชื่อสากล)
- **สั้น & เข้าใจง่าย:** ใช้คำศัพท์เทคนิคที่เป็นสากลและสั้นกระชับที่สุดเสมอ (เช่น `app`, `db`, `api`, `scripts`) หลีกเลี่ยงคำบรรยายยาวๆ เช่น `shared_scripts` หรือ `praem-os-ui`
- **รูปแบบ (Formatting):** 
  - ใช้ตัวพิมพ์เล็ก (lowercase) เท่านั้น
  - ห้ามเว้นวรรค ใช้เฉพาะ `snake_case` หรือ `kebab-case` เท่านั้น
- **กฎความปลอดภัยสูงสุด:** **ห้าม** ทำการเปลี่ยนชื่อไฟล์หรือโฟลเดอร์จำนวนมาก (Bulk Renaming) โดยไม่มีการวางแผน "ค้นหาและแทนที่ (Search & Replace)" เพื่ออัปเดตลิงก์ Markdown และสคริปต์ภายในทั้งหมดอย่างเด็ดขาด

## 2. The 3-Tier Systemic Memory Activation (ระบบเรียนรู้จากความผิดพลาด 3 ระดับ)
ระบบนี้เรียนรู้จากความล้มเหลว หาก Agent หรือกระบวนการทำงานล้มเหลว กลไกทั้ง 3 ระดับนี้จะต้องถูกเปิดใช้งานทันที:
1. **Tier 1 (Feedback Loop):** ความผิดพลาดทางสถาปัตยกรรม, ความพยายามที่ล้มเหลว, และการแก้ไขที่เกิดขึ้น "ต้อง" ถูกบันทึกลงใน `system/session_log.md` ทันที (ทั้งใน vault ฝั่ง English และ Thai)
2. **Tier 2 (SOP Playbooks):** กฎเกณฑ์เชิงโครงสร้างหรือพฤติกรรมที่เกิดจากความผิดพลาดจะต้องถูกร่างเป็นเพลย์บุ๊กนี้ (Playbook) หรือเพลย์บุ๊กใหม่ใน `system/skills/` โดย Agent ทุกตัวจะต้องปฏิบัติตามมาตรฐานเหล่านี้
3. **Tier 3 (HR Audits):** Chief People & Performance Officer (Grace) จะคอยตรวจสอบและประเมินกระบวนการทำงานของ Agent อย่างต่อเนื่อง Grace จะดูแลให้มีการปฏิบัติตามนโยบาย HITL (Human-in-the-loop), ความคุ้มค่าในการใช้ Token (ไม่มีการวนลูปที่สูญเปล่า), และความสอดคล้องกับ Naming Protocol
